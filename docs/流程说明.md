# 采集与蒸馏 — 流程说明（单一入口）

**用途**：统一说明「从多源采集 → 待蒸馏 → 蒸馏 → 待归属 → 迁移到知识库」的整条链路。适用于线上/云环境，**路径均由环境变量配置**，不依赖本地目录。细节与脚本用法见 `scripts/README.md`，归属目标见 `docs/蒸馏归属目标.md`。信息源统一维护在 `config/信息源列表.md`（每行一条）。

---

## 一、总览：一条线走到底

```
多源采集 → 待蒸馏/ → 批量蒸馏 → 蒸馏产出_待归属/ → 按规则归属并迁移 → about_*
```

| 阶段 | 输入 | 输出 | 关键脚本/配置 |
|------|------|------|----------------|
| **1. 采集** | 信息源列表（每行一个 URL） | `待蒸馏/*.md` + history 去重 | `collect_sources.py` |
| **2. 蒸馏** | `待蒸馏/` 中未在 history 的 .md | `蒸馏产出_待归属/YYYYMMDD_distill_NNN.md` + history | `distill_run.py`（单条或 `--target` 批量，DeepSeek） |
| **3. 归属与迁移** | `蒸馏产出_待归属/*.md` | about_* 下按主题落盘，待归属内文件删除 | `归属匹配规则.json`、`assign_and_migrate.py`、可选 `migrate_by_csv.py` |

**去重**：采集与蒸馏均写 `distill_input_history.json`（URL/路径/content_hash），同一内容不重复采集、不重复蒸馏。

---

## 二、数据流与目录

```
信息源（config/信息源列表.md，每行一个 URL）
├── RSS/Atom 订阅 URL
└── 主页面 URL
         │
         ▼  collect_sources.py
    待蒸馏/
         │
         ▼  distill_run.py --target N
    蒸馏产出_待归属/
         │
         ▼  assign_and_migrate.py（按 归属匹配规则.json）或 人工/CSV
    about_industry_analysis/、about_startup/、about_financials/、about_energy_storage/ 等
```

- **待蒸馏**：只放「尚未蒸馏」的原始/摘要内容，蒸馏后 history 有记录，不会再次被 batch 选中。
- **蒸馏产出_待归属**：只放「已蒸馏、尚未迁到 about_*」的产出；迁移后文件从该目录删除，并在 `蒸馏归属目标.md` 记一笔。

## 三、推荐动线

### 方案 A：多源采集 + 按目标条数蒸馏（推荐）

```bash
cd <项目根>/scripts
conda activate ikb_env
python collect_sources.py
python distill_run.py --target 50
export KB_VENTURE=<知识库根路径>
python assign_and_migrate.py --migrate
```

### 方案 B：只做归属/迁移（已有待归属文件时）

```bash
python assign_and_migrate.py --dry-run
python assign_and_migrate.py --migrate
```

## 四、易混点与约定

| 易混点 | 说明 |
|--------|------|
| **采集源** | 仅维护 `config/信息源列表.md`，每行一个 URL；运行 `collect_sources.py` 即处理全部行。 |
| **待蒸馏 vs 蒸馏产出_待归属** | 待蒸馏 = 未蒸馏的**输入**；蒸馏产出_待归属 = 已蒸馏的**输出**，等迁移。 |
| **归属谁决定** | 默认由 **config/归属匹配规则.json**（关键词 + 顺序）自动分配；可编辑该 JSON 或改生成的 CSV 后再执行迁移。 |
| **迁移脚本** | 统一用 **assign_and_migrate.py**；**migrate_by_csv.py** 仅在有现成 CSV 时做「按表迁移」。 |
| **路径** | 知识库根由 **KB_VENTURE** 提供（迁移时必设）；项目根可由 **DISTILL_PROJECT_ROOT** 指定，不在脚本中写死本地路径。 |
