# 任务体系设计经验总结：从知识蒸馏到知识库分配与任务体系优化

**用途**：记录「信息源蒸馏 → 知识库归属 → 任务体系设置优化」这一轮构建过程中的关键决策、做法与可复用经验，供后续任务体系设计改进时查阅。  
**关联**：`docs/架构.md`、`docs/蒸馏归属目标.md`、COORDINATION 维护清单。

---

## 一、构建过程总览

```
1. 知识蒸馏（提示与产出质量）
   → 2. 知识库分配（归属目标与待归属缓冲）
   → 3. 任务体系设置优化（维护清单、加载约定、规则补强）
```

| 阶段 | 主要动作 | 产出/约定 |
|------|----------|------------|
| **蒸馏** | 定义「一条知识单元」的深度、可泛化、支持任务决策；明确下游用途为「候选转给 LLM 做场景分析」 | 蒸馏提示词、SKILL 正文要求、列表/清单类保留规则 |
| **知识库分配** | 建立「可归属知识库」唯一列表；引入「蒸馏产出_待归属」缓冲目录；按主题迁移并记录 | 蒸馏归属目标.md、蒸馏产出_待归属/、迁移记录 |
| **任务体系优化** | 维护清单覆盖「蒸馏+归属+配置」全执行；知识库加载约定（文件夹内全部 .md）；规则强制「读 COORDINATION / 记 TASK_OUTCOMES」 | maintenance-full-execution、KB_PATHS 加载约定、coordination-and-logging |

---

## 二、各阶段关键决策与做法

### 2.1 知识蒸馏

- **做法**：将目标改为「**一条有深度的、可泛化的、能支持任务决策的** Markdown 知识单元」，并写明「这些知识单元会作为**候选**转给 LLM 做针对场景的分析」。清单/榜单/调查类：明确要求保留代表性条目或分类结构，不得仅一句概括。
- **教训**：**提示词要显式写出「下游用途」和「质量维度」**（深度、可泛化、支持决策），否则模型会默认往「简短摘要」收敛。

### 2.2 知识库分配

- **做法**：**单一权威列表**（蒸馏归属目标.md）；**缓冲目录**（蒸馏产出_待归属）；**迁移记录**；**归属边界**与用户确认哪些路径不接收蒸馏产出。
- **教训**：**归属目标单源化 + 缓冲目录 + 迁移记录**，可减少遗漏和歧义；命名要体现「临时/待处理」而非具体领域名。

### 2.3 任务体系设置优化

- **做法**：**维护清单全覆盖**；**知识库加载约定**（主文档 + 概念索引 + 该目录下除 README 外的全部 .md）；**调度与记录强制**。
- **教训**：**「做一次 X」类清单要显式列出所有子项并禁止半执行**；**加载约定要写清「读哪些文件」**，避免新知识入库却不可见。

---

## 三、可复用经验总结

1. **提示词与下游用途绑定**：蒸馏/抽取类提示要显式写出产出的下游用途和质量维度。
2. **清单/结构化内容在提示中单独约定**：原文含清单、榜单时须在提示中单独一条要求保留「代表性条目或分类结构」。
3. **归属目标单源化 + 缓冲 + 记录**：可写入的目录列表只维护在一处；产出先入缓冲目录，再按表迁移并记录迁移关系。
4. **「做一次 X」清单要全量、可检查**：维护清单应列出全部必做子项，并用规则禁止「只做一部分」。
5. **知识库加载约定要写清范围**：约定是「主文档+索引」还是「主文档+索引+目录下全部 .md」。
6. **命名避免与正式资产混淆**：临时/缓冲目录命名宜体现「待处理」「待归属」。
7. **规则与架构分层、不重复**：完整定义只在 COORDINATION / 蒸馏归属目标 / 架构文档等单处维护；规则文件只做引用与强制。
